Lighthouse

AI-Powered O-1 Visa Onboarding

An agentic prototype that replaces static forms with guided, AI-coached evidence collection

Welcome. Lighthouse is our answer to a core problem in immigration onboarding: applicants don't understand what good evidence looks like, and the current process of emails, spreadsheets, and phone calls doesn't scale. Today I'll walk through the problem, our solution, the AI architecture, and a live demo.

The Problem

O-1 Visa Onboarding Today

  • 8 possible USCIS criteria, dozens of evidence fields
  • Long text blocks + emails/calls — users don't read them
  • No feedback until after submission
  • 3-4 revision cycles per criterion on average

Why It Matters Now

  • H-1B fee tripling to $100K → explosive O-1 demand
  • 94% O-1A approval rate, but evidence assembly is the bottleneck
  • Manual intake doesn't scale with growing caseloads
The core pain: applicants are high-achieving but immigration-naive. They receive a spreadsheet of required documents, spend weeks gathering evidence without knowing if it's sufficient, submit, get rejected, and repeat 3-4 times per criterion. That's the loop we're breaking.

The Key Insight

Assess evidence quality during collection, not after.

This is the single most important design decision. Every other tool evaluates evidence after the fact. We evaluate it inline, while the applicant is writing. That means they can improve in real-time instead of waiting days for feedback.

Pydantic Schema

EvidenceGrade {
  verdict: str
  score: float (0-1)
  feedback: str
  suggestions: list[str]
}

Why Structured Output?

Guarantees verdict is one of 3 values, score is 0-1 float. No regex parsing. Downstream UI + outreach depend on clean data.

Why Mock Mode?

Full demo without API costs. Length heuristic: >200 chars = sufficient, >50 = needs improvement.

The grading pipeline uses LangGraph with a single node and Pydantic structured output. The LLM is constrained to return a specific schema — no free-form text to parse. This is critical because the outreach engine and UI rendering both depend on clean verdict/score data. Mock mode activates when no API key is set. It uses text length as a grading heuristic — surprisingly effective for demos since longer, more detailed evidence actually does tend to be better. Why LangGraph for a single node? Extensibility. Adding document parsing or salary benchmarking is just adding a new node — no architecture changes.

Immediate after review:

  • approval notifications for strong criteria

After 24h (simulated):

Rule-based, no LLM — verdicts encode all needed info. Instant and deterministic.

The bonus requirement was handling pushback — where a case manager rejects insufficient evidence. Our system does this automatically. After AI review, each criterion gets a verdict. Weak criteria trigger reminder emails, and the user can jump back to improve specific fields without losing any prior work. The outreach engine is rule-based, not LLM-powered. Verdicts already contain all the information needed to decide what action to take.

Assumptions & Tradeoffs

Assumptions Made

  • Case strategy JSON format: {criteria: [{name, description, fields}]}
  • AI grading is coaching, not legal decision-making
  • Text evidence is primary review target (files stored, not parsed)
  • Single-user prototype (no auth, no persistence)
  • Mock mode is acceptable for API-key-free demos

What I'd Do Differently

  • Document parsing — PDF/image → vision LLM for evidence extraction
  • Persistent storage — DB-backed case data across sessions
  • Multi-node pipeline — Separate parsing, grading, fact-checking nodes
  • Attorney dashboard — Case manager oversight view
  • Salary benchmarking — DOL prevailing wage API for High Remuneration
Being transparent about assumptions. The biggest one: text evidence is what we grade, files are stored but not analyzed. Document parsing via vision LLM is the natural next step and fits cleanly into the existing LangGraph pipeline — just add a new node. The architecture was designed to make this extension easy.
