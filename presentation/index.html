Welcome. Lighthouse is our answer to a core problem in immigration onboarding: applicants don't understand what good evidence looks like, and the current process of emails, spreadsheets, and phone calls doesn't scale.

The Problem

O-1 Visa Onboarding Today

  • 8 possible USCIS criteria, dozens of evidence fields
  • Long text blocks + emails/calls — users don't read them
  • No feedback until after submission
  • 3-4 revision cycles per criterion on average

Why It Matters Now

  • H-1B fee tripling to $100K → explosive O-1 demand
  • 94% O-1A approval rate, but evidence assembly is the bottleneck
  • Manual intake doesn't scale with growing caseloads
The core pain: applicants are high-achieving but immigration-naive. They receive a spreadsheet, spend weeks gathering evidence without knowing if it's sufficient, submit, get rejected, and repeat 3-4 times per criterion.

The Key Insight

Assess evidence quality during collection, not after.

This is the single most important design decision. Every other tool evaluates evidence after the fact. We evaluate it inline — applicants improve in real-time instead of waiting days for feedback.

Pydantic Schema

EvidenceGrade {
  verdict: str
  score: float (0-1)
  feedback: str
  suggestions: list[str]
}

Why Structured Output?

Guarantees verdict is one of 3 values, score is 0-1 float. No regex parsing. Downstream UI + outreach depend on clean data.

Why Mock Mode?

Full demo without API costs. Length heuristic: >200 chars = sufficient, >50 = needs improvement.

LangGraph with a single node and Pydantic structured output. The LLM returns a typed schema — no free-form text to parse. Mock mode uses text length as a grading heuristic. Why LangGraph for one node? Adding document parsing or salary benchmarking is just a new node.

Assumptions & Tradeoffs

Assumptions Made

  • Case strategy JSON: {criteria: [{name, description, fields}]}
  • AI grading is coaching, not legal decision-making
  • Text evidence is primary review target (files stored, not parsed)
  • Single-user prototype (no auth, no persistence)
  • Mock mode acceptable for API-key-free demos

What I'd Do Differently

  • Document parsing — vision LLM for PDF/image evidence
  • Persistent storage — DB-backed case data
  • Multi-node pipeline — parsing, grading, fact-checking
  • Attorney dashboard — case manager oversight
  • Salary benchmarking — DOL prevailing wage API
Biggest assumption: text evidence is what we grade, files stored but not analyzed. Document parsing via vision LLM is the natural next step — just add a LangGraph node.
